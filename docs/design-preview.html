BetterBudgeter — Design Language

An evidence-based, ADHD-optimized personal finance interface. Every design decision is grounded in empirical research on cognitive accessibility (W3C COGA, Neurodiversity Design System, British Dyslexia Association). The system anchors on a warm cream canvas with desaturated semantic colors — soft coral replaces red to avoid financial anxiety, soft green signals progress without overstimulation.

Calm over comprehensive Compassion over correction Clarity over cleverness Consistency over customization Progressive over prescriptive

01 · Overview

Calm, clarity, compassion #

BetterBudgeter is a personal finance tracker for users whose executive function may be impaired by ADHD, anxiety, or cognitive fatigue — conditions that conventional budgeting apps actively worsen through information overload, judgmental language, and anxiety-inducing color choices.

Design Principles

Five principles guide every design decision, in priority order. When two principles conflict, the lower-numbered one wins.

P1

Calm Over Comprehensive

Show only what the user needs right now. Every element must earn its place on screen. The goal is a financial tool that feels calm, not overwhelming.

W3C COGA §4.6.3
P2

Compassion Over Correction

Never make the user feel bad about their financial decisions. Frame setbacks as normal, celebrate progress quietly, and treat overspending as information — not failure.

Shame drives abandonment
P3

Clarity Over Cleverness

Prefer explicit, readable patterns over compact, clever ones. A stressed user should understand the interface without explanation.

W3C COGA §4.3
P4

Consistency Over Customization

The same pattern should look and behave the same everywhere. Predictability reduces cognitive load.

W3C COGA §4.2
P5

Progressive Over Prescriptive

Start simple, let users discover complexity. Smart defaults for the common case. Advanced options available but never required.

ADHD UX research

Visual Identity

The visual language is warm, quiet, and structured. A subtle cream canvas (inspired by the humanist warmth of Anthropic's Claude design language) replaces the sterile white of typical fintech apps, reducing eye strain and creating a sense of comfort rather than clinical efficiency. Colors are intentionally desaturated — soft coral instead of alarm-red, muted amber instead of traffic-light yellow. The overall impression should be a patient friend who's good with money, not a stern financial advisor.

02 · Colors

Color system #

Authoritative color values are in oklch. The hex strings are sRGB approximations for tooling compatibility. oklch provides perceptually uniform lightness — equal numeric steps produce equal visual changes — making it ideal for systematic accessible palette design.

Canvas & Surface

Warm light surfaces reduce cognitive fatigue during daytime use (NDS recommendation), while cool dark surfaces reduce eye strain at night. The asymmetry is intentional.

Text

Text colors maintain a cool tint (hue 280°) for high contrast against the warm canvas. Primary text exceeds WCAG 2.2 AA (4.5:1) on both cream and white surfaces.

Semantic Colors (Financial)

These colors carry consistent meaning across the app. They are the emotional backbone of the interface. We avoid bright red because research on color psychology in financial contexts shows that red triggers anxiety and shame responses (Elliot & Maier, 2014). Our "negative" is a soft coral — same hue family but reduced chroma.

Budget Category Colors

All category colors sit at similar lightness values (0.58–0.68) so no single category dominates a chart. Chroma is held low (0.02–0.14) for a calm overall palette.

03 · Typography

Typography #

BetterBudgeter uses the system sans-serif stack exclusively. The British Dyslexia Association recommends sans-serif typefaces with clear character differentiation. System fonts meet this requirement while loading instantly and feeling native to each platform.

Font Stack

--bb-font-sans:
  "Lucida Sans Unicode", "Lucida Grande", "Lucida Sans", Verdana,
  -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto,
  "Helvetica Neue", Arial, sans-serif;

--bb-font-mono:
  ui-monospace, "SF Mono", "Cascadia Code", Menlo, monospace;

Type Scale

Every sample is rendered at its real size, weight, line-height, and tracking. Minimum body size is 16px — a hard constraint, not a guideline.

Rules

  • Body text never below 16px.
  • Line height ≥ 1.5 for body (we use 1.6).
  • Positive letter-spacing for body (+0.01em) and captions (+0.02em).
  • Max line length 65–75 characters (max-width 768px).
  • Emphasis via semibold (600) or color — never italics.
  • Left-aligned only — never justified.

04 · Layout

Layout & spacing #

Built on a 4px grid with a primary rhythm of 8px. W3C COGA §4.4.10 explicitly recommends generous whitespace for users with cognitive disabilities. Whitespace is a cognitive accessibility tool, not wasted space.

Spacing Scale

Layout Constraints

Property Value Rationale
Max content width 768px Prevents lines exceeding 75 characters
Page padding 16px mobile, 24px tablet+ Breathing room at edges
Card padding 20px Generous internal space
Card gap 16px Clear separation between cards
Section gap 32px Visual breathing room between sections

Layout Patterns

Single-column (default). All pages use a single-column layout with stacked cards. This is the most readable pattern and works identically on mobile and desktop. Multi-column dashboards create cognitive overload.

Two-column grid (limited use). Only for paired summary cards (e.g., Income + Expenses side-by-side). Stacks to single column on mobile (< 640px).

Page Architecture

Hub-and-spoke navigation with a persistent bottom tab bar (4 tabs: Home, Budgets, Transactions, Settings). Research supports 3–5 navigation items for ADHD users — staying within working memory limits. Each page has a maximum of 4 content sections.

05 · Elevation

Elevation & depth #

BetterBudgeter uses a deliberately flat elevation system. Depth comes from background color shifts and subtle borders, not heavy shadows. The warm cream canvas creates a natural "depth" effect — white cards appear to float above the cream background without needing visible shadows.

06 · Shapes

Shapes & radii #

Rounded shapes are perceived as friendlier and less complex than sharp corners (Bar & Neta, 2006). In a financial context — where anxiety is inherent — softer radii help counteract the emotional weight of money management.

08 · Do's & Don'ts

Do's & Don'ts #

Five categories, paired guidance. Every "Don't" has an explicit "Do" alternative — never leave the reader without a path forward.

Color

✓ Do

Use soft coral (--bb-negative) for overspending indicators.

Red triggers anxiety responses in financial contexts (Elliot & Maier, 2014).

× Don't

Use bright red, even for critical errors.

Soft coral at hue 25° with chroma 0.16 communicates urgency without panic.

✓ Do

Use semantic background tints to communicate status at a glance.

Color reinforces meaning when paired with text.

× Don't

Rely on color alone — always pair color with text labels.

WCAG 2.2: Use of Color.

Typography

✓ Do

Maintain 16px minimum for body text. Use 12px only for timestamps and captions.

W3C COGA + NDS hard constraint.

× Don't

Use italics for emphasis. Use semibold (600) or semantic color instead.

British Dyslexia Association guideline.

✓ Do

Left-align all text.

Justified text creates uneven spacing that disrupts reading flow.

× Don't

Use justified or center-aligned body text.

BDA, NDS.

Layout

✓ Do

Limit each page to 4 content sections maximum.

7+ sections create cognitive overload for ADHD users.

× Don't

Create multi-column dashboard layouts.

Single-column reads top-to-bottom like a document.

✓ Do

Use generous whitespace between sections (32px+).

W3C COGA §4.4.10 — whitespace is a cognitive accessibility tool.

× Don't

Pack elements tightly to "fit more on screen."

Visual crowding increases parsing load.

Language

✓ Do

Frame overspending as information: "Your Food spending reached €200."

Neutral framing avoids shame responses.

× Don't

Use judgmental language: "You exceeded your budget" or "Budget failed."

Shame is the primary driver of financial app abandonment.

✓ Do

Lead with what the user can do: "Adjust your budget or review recent spending."

A path forward is more useful than a verdict.

× Don't

Use streaks, achievements, or gamification.

Counterproductive for ADHD users with inconsistent patterns.

Motion

✓ Do

Respect prefers-reduced-motion: reduce for all non-essential animation.

WCAG 2.2 + ADHD research.

× Don't

Use bouncing, pulsing, parallax, or looping animations.

Motion sensitivity and attention disruption.

✓ Do

Keep transitions under 300ms.

Long animations feel like blockers, not feedback.

× Don't

Auto-dismiss toasts or notifications.

W3C COGA §4.6.1 — respect the user's reading speed.

Interaction

✓ Do

Use confirmation dialogs with supportive language: "Remove this budget? You can recreate it anytime."

Explain what happens, not what could go wrong.

× Don't

Ask "Are you sure?" with no further context.

Adds anxiety without adding information.

✓ Do

Show skeleton screens for page loads.

Maintains layout; avoids spinner-only blocking.

× Don't

Block the entire page with a loading spinner.

Layout shift on completion increases cognitive load.

09 · Motion

Motion & animation #

All non-essential motion respects prefers-reduced-motion: reduce. Animations are functional (communicate state change), never decorative. Maximum duration 300ms for transitions, 500ms for complex sequences. Easing: ease-out for entering elements, ease-in for exiting.

Allowed animations

Card appearFade + subtle rise (0→4px translateY)200ms
Progress bar fillWidth transition300ms

Toast exitFade out150ms

Prohibited animations

  • Autoplay animations on page load (except skeleton screens).
  • Bouncing or pulsing elements.
  • Parallax scrolling.
  • Background animations.
  • Animated gradients.
  • Any animation that loops indefinitely.

10 · Copy

Copy & language #

Supportive. Like a patient friend who's good with money — not a financial advisor, not a parent, not a teacher. Neutral, warm, informational.

Rules

  • Never blame. No "you overspent," "budget failed," or "you need to."
  • Neutral framing. "Your Food spending reached €200" instead of "You exceeded your Food budget."
  • Lead with action. "Adjust your budget or review recent spending" instead of just reporting the problem.
  • Avoid jargon. "Spending limit" instead of "budget allocation."
  • Be specific. "€30 remaining in Food" instead of "Budget is getting low."
  • Celebrate quietly. "All budgets on track this month" — no elaborate achievement systems.

Specific rewrites

Context Avoid Use instead
Over budget "Budget exceeded" "You've used your full Food budget. €20 over."
Warning "Watch your spending" "Food is at 85%. €30 left for this month."
On track "You're doing well" "All budgets on track."
No data "No transactions found" "Transactions will appear here after your first sync."
Error "Failed to sync" "Couldn't connect to your bank. Try again?"
Empty budget "No budgets set" "Set up a budget to start tracking your spending."

11 · Accessibility

Accessibility #

BetterBudgeter targets WCAG 2.2 AA at minimum, with additional ADHD-specific requirements layered on top.

WCAG 2.2 AA

  • 4.5:1 contrast for body text, 3:1 for large text and UI components.
  • Touch targets minimum 44×44px.
  • Visible focus ring on all interactive elements.
  • All interactive elements reachable via Tab key.
  • Proper ARIA labels, semantic HTML, landmark regions.
  • App remains functional at 200% zoom.

ADHD-specific

  • prefers-reduced-motion — disable non-essential animations.
  • prefers-color-scheme — support both light and dark.
  • No time limits — no auto-dismissing toasts.
  • Consistent layout — no layout shifts during loading.
  • Single-column reading order — top to bottom.
  • Max line length 65–75 characters.

Semantic HTML

  • Use <main>, <nav>, <header>, <section> landmarks.
  • Headings follow strict hierarchy (h1 → h2 → h3, no skipping).
  • Form inputs always have associated <label> elements.
  • Interactive elements use <button> or <a>, never <div> with onClick.
  • Error messages linked to inputs via aria-describedby.
