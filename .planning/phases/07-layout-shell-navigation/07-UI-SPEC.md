## Spacing Scale

Declared values (Phase 6 tokens, 4px grid — already wired):

| Token | Value | Tailwind utility | Usage in Phase 7 |
|-------|-------|------------------|------------------|
| `--bb-space-1` | 4px | `p-bb-1`, `gap-bb-1`, `mt-bb-1` | Icon-to-label gap inside TabBar tab (label hugs icon) |
| `--bb-space-2` | 8px | `p-bb-2`, `gap-bb-2`, `mt-bb-2` | PageHeader: gap between title and subtitle (DESIGN_SYSTEM §3.3 — "headings followed by 8px") |
| `--bb-space-3` | 12px | `p-bb-3` | (reserved for spoke content; no direct use this phase) |
| `--bb-space-4` | 16px | `px-bb-4` | PageShell horizontal padding on mobile (DESIGN_SYSTEM §4.2) |
| `--bb-space-5` | 20px | `p-bb-5` | Card internal padding (locked, used by Phase 8/9 cards; reserved here) |
| `--bb-space-6` | 24px | `px-bb-6`, `pt-bb-6` | PageShell horizontal padding ≥768px (`md:px-bb-6`); PageShell top padding |
| `--bb-space-8` | 32px | `mb-bb-8`, `gap-bb-8` | Section gap below PageHeader; section-to-section gap inside spoke pages |
| `--bb-space-10` | 40px | — | (unused in Phase 7) |
| `--bb-space-12` | 48px | — | (unused in Phase 7) |

**Exceptions for Phase 7:**

- **TabBar height — 56px (literal, not a token).** Sits between `--bb-space-12` (48px) and the un-tokenized 64px step. 56px is a standard mobile bottom-nav height (matches iOS/Material defaults) and provides 44px+ touch targets after subtracting 4×4=8px internal padding. Encoded as a literal in `TabBar.tsx` (`h-[56px]`) AND mirrored in `PageShell.tsx`'s bottom-padding formula. Document this in component header comments so the two stay in sync.
- **Touch target — 44×44px minimum per tab** (`min-w-[44px] min-h-[44px]` per A11Y-01 / NAV-02). Achieved within the 56px bar via icon (22px) + 4px gap + label (12px line-height) + ~4px vertical padding.
- **PageShell bottom padding formula — `pb-[calc(56px+env(safe-area-inset-bottom)+1rem)]`.** Sum of: TabBar height (56px) + iOS home-indicator clearance (`env(safe-area-inset-bottom)`, typically 34px on notched iPhones) + 1rem (16px) comfort gap so content can fully scroll past the bar. Hardcoded literal because none of the existing spacing tokens compose to this exact sum.

## Typography

Declared values (Phase 6 type-scale tokens — already wired). **5 sizes, 2 weights** per ADHD minimalism principle (TOKEN-07).

| Role | Token | Size | Weight | Line Height | Tailwind class | Used by |
|------|-------|------|--------|-------------|----------------|---------|
| Caption (TabBar label) | `--bb-text-xs` | 12px | 400 | 1.5 | `text-bb-xs` | TabBar tab labels |
| Stub body / secondary | `--bb-text-sm` | 14px | 400 | 1.5 | `text-bb-sm` | Reserved (not used in Phase 7) |
| Body / subtitle | `--bb-text-base` | 16px | 400 | 1.6 | `text-bb-base` | PageHeader subtitle, stub-page placeholder copy |
| Section heading | `--bb-text-xl` | 22px | 700 | 1.4 | `text-bb-xl` | Reserved (Phase 8/9 cards) |
| Page title (h1) | `--bb-text-3xl` | 36px | 700 | 1.2 | `text-bb-3xl` | PageHeader title |

**Phase 7-specific typography rules:**

- **PageHeader title:** `<h1>` element, `text-bb-3xl`, `font-bold` (weight 700, paired with the 36px size per TOKEN-07), color `text-bb-text`. Line-height 1.2 comes via the `--bb-text-3xl-lh` companion token.
- **PageHeader subtitle:** `<p>` element, `text-bb-base`, weight 400 (default), color `text-bb-text-secondary`, top margin `mt-bb-2` (8px gap below title per DESIGN_SYSTEM §3.3).
- **TabBar label:** `<span>` inside the `<Link>`, `text-bb-xs` (12px, weight 400), color matches tab state (`text-bb-info` active / `text-bb-text-secondary` inactive — see §Color). Use `leading-none` to keep the label visually tight under the icon.
- **Stub page body:** `<p>` element, `text-bb-base`, weight 400, color `text-bb-text-secondary` (secondary text — explicitly NOT primary text, signals placeholder state to the user without using dashed boxes or icons per D-08).
- **Alignment rule (DESIGN_SYSTEM §3.3):** All text in Phase 7 components is **left-aligned**. No centered text. No justified text.
- **Italics rule (DESIGN_SYSTEM §3.3):** Never use italics for emphasis in Phase 7 components. Use color or weight if emphasis is needed (none is in Phase 7).
- **Max line length (DESIGN_SYSTEM §3.3):** PageShell's 768px max-width inherently keeps body lines under the 65–75 char limit at 16px body size. No further per-component constraints needed.

**Anti-patterns (must NOT be present in Phase 7 implementation):**

- ❌ Hardcoded hex/rgb/oklch values in component files (A11Y-04 — token-only).
- ❌ Use of `text-red-500`, `bg-blue-600`, or any Tailwind default palette (defeats the calm desaturated tone; tokens only).
- ❌ Bright red anywhere (`--bb-negative` is soft coral by design — see DESIGN_SYSTEM §2.1 rationale).
- ❌ Background tints on the TabBar surface (use flat `bg-bb-surface`, no `bg-bb-info-bg` or similar).
- ❌ Filled background on the active tab pill (active state is icon-fill + text-color only; no background change).

## Responsive Behavior

| Breakpoint | Behavior |
|------------|----------|
| <640px (mobile) | PageShell: 16px horizontal padding. TabBar: full viewport width (because `w-full` caps at `max-w-[768px]` which is wider than the viewport). Single-column content (default for all pages in this milestone per DESIGN_SYSTEM §4.3). |
| ≥640px and <768px (large mobile / small tablet) | PageShell still 16px horizontal padding (Tailwind `md:` breakpoint is 768px in this project per Tailwind v4 defaults). TabBar still fills viewport (still under 768px). |
| ≥768px (tablet+, including desktop) | PageShell: 24px horizontal padding (`md:px-bb-6`), 768px max-width centered. TabBar: 768px wide, centered at viewport bottom (D-09 — reinforces PWA/mobile-app feel even on desktop). Empty space on either side of TabBar shows the `--bb-bg` page background. |

**Mobile-vs-desktop deliberate sameness:** The TabBar and PageShell render the **same component tree at all breakpoints**. The only changes across breakpoints are CSS-driven padding/width tweaks. No "hide on desktop" or "show as sidebar on large screens" patterns. This is intentional per P4 (consistency) and CONTEXT §Specifics ("TabBar feels like a mobile app on desktop too — user explicitly chose the 768px-constrained centered TabBar over a full-width desktop nav, prioritizing consistency with the PWA/ADHD design language").

## Anti-Patterns to Flag During Implementation

If the executor or auditor encounters any of these, treat as a contract violation:

1. ❌ Hardcoded color values (hex/rgb/oklch) in any Phase 7 file. **Token-only** (A11Y-04).
2. ❌ Use of Tailwind default palette utilities like `text-blue-500`, `bg-zinc-100`, `border-gray-200`. Use `text-bb-*`, `bg-bb-*`, `border-bb-*`.
3. ❌ Direct `@radix-ui/*` imports in any new Phase 7 component (CLAUDE.md UI Library Boundaries — Radix is legacy-only).
4. ❌ Different layout / different navigation on mobile vs desktop (breaks P4 + D-09).
5. ❌ Animation on tab switch (transitions, fades, slides between routes — NAV-03 prohibits).
6. ❌ Touch target <44×44px on any TabBar tab (NAV-02 + A11Y-01).
7. ❌ Missing `aria-current` on active tab link (A11Y-03).
8. ❌ Multiple `<main>` elements per route tree (RESEARCH Pitfall 5).
9. ❌ Missing `viewportFit: "cover"` in root layout's Viewport export (RESEARCH Pitfall 1 — TabBar will sit under iOS home indicator).

## Open Questions for Checker

These are surfaced for `gsd-ui-checker` to verify or escalate; they are not blockers for the contract.

1. **Active-tab label contrast at 12px.** `--bb-info` on `--bb-surface` should be measured for 4.5:1 against the 12px label specifically (body-text rule, not large-text rule). If it fails, options are (a) bump label to 14px (`text-bb-sm`), (b) darken `--bb-info` (touches Phase 6 tokens — last resort), or (c) accept and rely on icon-fill + color combination as the active signal with the label as supporting reinforcement. Recommend (a) if checker flags the failure. RESEARCH §Color Contrast Verification.

2. **Lucide filled-icon visual quality.** `fill="currentColor" strokeWidth={0}` on House / ChartPie / List / Settings needs visual confirmation in browser. If any icon looks like a black blob (Pitfall 3), checker should recommend an alternative lucide icon (e.g. `Home` instead of `House`, `Menu` instead of `List`, `Cog` instead of `Settings`). RESEARCH §Open Questions 2.

3. **PageShell `<main>` placement vs existing pages.** RESEARCH Pitfall 5 notes that the current `src/app/page.tsx` (Home) has its own `<main>`. When that page is moved into `(bb)/`, its wrapper must be downgraded to `<div>` or `<section>` to avoid double-`<main>`. Verifier should confirm this is handled by the planner/executor and not leaked into the rendered HTML.

4. **Focus-ring visibility on `--bb-surface`.** The default global ring `outline-ring/50` from `@layer base` may have weak contrast on `--bb-surface` in light mode. Checker should test keyboard-tab focus on the TabBar in both modes. If insufficient, add explicit `focus-visible:ring-2 focus-visible:ring-bb-info focus-visible:ring-offset-2` per A11Y-01.
