| Token | Value | Usage in Phase 8 |
|-------|-------|------------------|
| `--bb-space-1` | 4px | Icon-to-text gap (traffic-light dot to category name) |
| `--bb-space-2` | 8px | Label-to-number gap inside hero block; heading bottom margin |
| `--bb-space-3` | 12px | TransactionItem secondary line top offset |
| `--bb-space-4` | 16px | Card gap between cards within a section; horizontal card padding mobile |
| `--bb-space-5` | 20px | Card internal padding (all section cards) |
| `--bb-space-6` | 24px | PageShell horizontal padding tablet+ |
| `--bb-space-8` | 32px | Section gap — vertical distance between the 4 home hub sections |
| `--bb-space-10` | 40px | (available; not used on home hub in Phase 8) |
| `--bb-space-12` | 48px | (available; not used on home hub in Phase 8) |

**Exceptions:**
- TabBar bottom clearance: `pb-[calc(56px+env(safe-area-inset-bottom)+1rem)]` — set by PageShell (Phase 7); home page does not repeat this.
- Traffic-light dot: 8px diameter (2×2 in Tailwind `w-2 h-2`) — below the 4px grid minimum but justified as a decorative/semantic indicator only, not a touch target.

## Typography

All sizes from the `--bb-text-*` token system (Phase 6). Tailwind form: `text-bb-{name}`.

Note: globals.css implements 5 sizes (xs, sm, base, xl, 3xl) — the full DESIGN_SYSTEM.md scale also lists lg and 2xl but those were not wired in Phase 6. Phase 8 uses only the 5 confirmed wired sizes.

| Role | Token | Size | Weight | Line Height | Phase 8 Usage |
|------|-------|------|--------|-------------|---------------|
| Caption / timestamp | `text-bb-xs` | 12px | 400 | 1.5 | TransactionItem date; budget row remaining amount |
| Label / secondary | `text-bb-sm` | 14px | 400 | 1.5 | TransactionItem category label; budget row category name; "Safe to spend this month" label |
| Body | `text-bb-base` | 16px | 400 | 1.6 | Greeting text ("Good morning, Paul."); inline empty-state sentences; DB error note |
| Section heading | `text-bb-xl` | 22px | 700 | 1.4 | "Budget status" section heading; "Recent transactions" section heading |
| Hero display | `text-bb-3xl` | 36px | 700 | 1.2 | Safe-to-Spend EUR number (DESIGN_SYSTEM §5.1, CONTEXT D-CMP-01) |

**Rules (from DESIGN_SYSTEM §3.3):**
- No italics for emphasis — use `font-bold` or color.
- Always left-aligned. Never `text-justify`.
- Maximum ~70 chars per line — PageShell max-width 768px + card padding enforces this.

## Color

All via `--bb-*` tokens. No hardcoded hex or RGB. Both light and dark mode values are defined in globals.css.

| Role | Token | Tailwind Utility | Light Value | Phase 8 Usage |
|------|-------|-----------------|-------------|---------------|
| Dominant (60%) | `--bb-bg` | `bg-bb-bg` | `oklch(0.985 0.002 280)` | PageShell background (set by PageShell — home page inherits) |
| Secondary (30%) | `--bb-surface` | `bg-bb-surface` | `oklch(1 0 0)` | All section cards (hero, budget list, tx list, sync button area) |
| Raised surface | `--bb-surface-raised` | `bg-bb-surface-raised` | `oklch(0.975 0.002 280)` | Full-screen "link your bank" CTA card background |
| Primary text | `--bb-text` | `text-bb-text` | `oklch(0.205 0.006 280)` | All primary text; Safe-to-Spend number (always neutral — see hero color rule) |
| Secondary text | `--bb-text-secondary` | `text-bb-text-secondary` | `oklch(0.48 0.012 280)` | Greeting; all labels; category names; dates; empty-state body copy |
| Border | `--bb-border` | `border-bb-border` | `oklch(0.91 0.004 280)` | All card borders; budget row dividers |
| Accent / info | `--bb-info` | `text-bb-info`, `bg-bb-info-bg` | `oklch(0.60 0.12 245)` | Sync button (Primary variant); active tab (TabBar — already done Phase 7) |
| Positive | `--bb-positive` | `text-bb-positive` | `oklch(0.62 0.14 155)` | TransactionItem income amount; traffic-light dot (on-track, <80%) |
| Caution | `--bb-caution` | `text-bb-caution` | `oklch(0.72 0.14 75)` | Traffic-light dot (warning, 80–99%) |
| Negative | `--bb-negative` | `text-bb-negative` | `oklch(0.58 0.16 25)` | TransactionItem expense amount; traffic-light dot (over budget, ≥100%) |

**Accent reserved for:**
- The single Sync Transactions primary button (one per view — DESIGN_SYSTEM §5.6)
- Traffic-light dots use semantic colors (positive/caution/negative), NOT `--bb-info`

**Hero color rule — Safe-to-Spend number:**
Always `text-bb-text` (neutral dark), regardless of the computed value. Rationale: the formula clamps at 0 (no negatives are possible), and applying color coding to the hero number would add anxiety-inducing meaning that doesn't exist in the underlying math for this app's scope. Calm over comprehensive (P1).

**Dark-mode parity:**
All `--bb-*` tokens have confirmed dark-mode counterparts in globals.css. Traffic-light dots use the same semantic color tokens which shift to their lighter dark-mode variants automatically. The DB-error inline note uses `text-bb-text-secondary` — verified readable at 4.5:1 equivalent in both modes via token values.

## Component Visual Contracts

### Section 1: Greeting + Safe-to-Spend Hero

**Layout:** Vertical stack inside a `bg-bb-surface border border-bb-border rounded-bb-lg` card with `p-bb-5` internal padding.

**Structure (top to bottom):**
1. Greeting line — `text-bb-base text-bb-text-secondary` — e.g. "Good morning, Paul." Left-aligned.
2. Label — `text-bb-sm text-bb-text-secondary` — "Safe to spend this month" — left-aligned, `mt-bb-8` below greeting.
3. EUR number — `text-bb-3xl font-bold text-bb-text` — e.g. "€1,234.56" — left-aligned, `mt-bb-2` below label.

**Rationale for "Safe to spend this month" label positioning (label above number):**
Consistent with DESIGN_SYSTEM §5.1 KPI Card pattern: label → hero number → description. The label names the metric before the number is read, which aids ADHD users who scan top-to-bottom.

**No supporting description below the number** on Home. The description slot from §5.1 is omitted — Home is a glance view; description is available on the Transactions spoke page (Phase 9) if needed.

**No card-level background tint** (no `bg-bb-positive-bg` or equivalent) on the hero. Always neutral `bg-bb-surface`. Coloring the hero based on value would introduce anxiety signals the DESIGN_SYSTEM explicitly avoids (P1, P2). Tint variants belong on the Transactions page's Income/Expense KPI cards.

**`€—` state:** When `safeToSpend` is unavailable (DB error) render the string "€—" in `text-bb-text-secondary` at `text-bb-3xl`. Same layout, different color to signal missing data without visual alarm.

### Section 2: Budget Status Indicators

**Layout:** `bg-bb-surface border border-bb-border rounded-bb-lg p-bb-5` card, containing:
- Section heading: `<h2 className="text-bb-xl font-bold text-bb-text mb-bb-4">` — "Budget status"
- List of `BudgetStatusRow` items, each on one line.

**BudgetStatusRow visual contract (private function, one per active budget):**

```
● Food                              €80 remaining
```

- Traffic-light dot: `w-2 h-2 rounded-full flex-shrink-0` with color class:
  - `bg-bb-positive` — status `on_track` (< 80% used)
  - `bg-bb-caution` — status `warning` (80–99% used)
  - `bg-bb-negative` — status `over_budget` (≥ 100% used)
- Dot diameter: 8px (`w-2 h-2`). See spacing exceptions note.
- Category name: `text-bb-sm text-bb-text` — left-aligned, `ml-bb-2` from dot.
- Remaining amount: `text-bb-xs text-bb-text-secondary` — right-aligned, pushed by `flex-1` between name and amount.
- Row: `flex items-center gap-bb-2 py-bb-2` — no border, no background tint; plain rows separated only by vertical padding.
- Sort order: Alphabetical by category name (CONTEXT D-04 / Claude's Discretion).
- Rows are NOT clickable in Phase 8. No hover state. Plain `<div>` (Phase 9 owns the budget spoke destination).

**0-budgets empty state (inline, within card):**
- Replace list with: `<p className="text-bb-sm text-bb-text-secondary">Set up your first budget</p>` followed by `<a href="/settings" className="text-bb-sm text-bb-info underline-offset-4 hover:underline mt-bb-1 inline-block">Go to settings</a>`.
- No dashed border or icon — inline link is sufficient per Phase 8 scope. Phase 10 will refactor to the EmptyState primitive with dashed card.

### Section 3: Recent Transactions

**Layout:** `bg-bb-surface border border-bb-border rounded-bb-lg p-bb-5` card, containing:
- Section heading: `<h2 className="text-bb-xl font-bold text-bb-text mb-bb-4">` — "Recent transactions"
- List of `<TransactionItem>` components (3–5 items, `getRecentTransactions(5)`).

**TransactionItem visual contract (`src/components/dashboard/TransactionItem.tsx`):**

```
REWE Supermarkt                    −€34.50
Food · 14 Apr
```

- Outer element: `<div className="flex flex-col gap-bb-1 py-bb-3">` — consistent height via vertical padding.
- Row 1 (primary): `flex items-baseline justify-between`
  - Merchant name: `text-bb-base font-medium text-bb-text` — left.
  - Amount: `text-bb-base font-medium` — right; color by type:
    - `type === 'income'`: `text-bb-positive` + rendered as `+€XX.XX`
    - `type === 'expense'`: `text-bb-negative` + rendered as `−€XX.XX` (use `−` U+2212 minus sign, not hyphen-minus `-`)
- Row 2 (secondary): `flex items-center gap-bb-1`
  - Category: `text-bb-xs text-bb-text-secondary`
  - Separator: `·` (U+00B7 middle dot) with `text-bb-text-secondary`
  - Date: `text-bb-xs text-bb-text-secondary` — formatted `"14 Apr"` via `new Date(date).toLocaleDateString('de-DE', { day: 'numeric', month: 'short' })`
- No icons (DESIGN_SYSTEM §5.3, PAGE-03).
- No background on each row. Rows are separated by `border-b border-bb-border` on all but the last item (`last:border-b-0`).

**0-transactions empty state (inline, within the card):**
- Replace list with: `<p className="text-bb-sm text-bb-text-secondary mb-bb-4">Your transactions will appear here.</p>` followed by the `<SyncTransactionsButton accountCount={...} />` inline.
- Rationale: CONTEXT D-CUT-02 specifies this exact pattern. Sync button as the CTA prevents a dead-end empty state.

### Section 4: Sync Button

**Layout:** No card wrapper. The button sits directly in the page flow, `mt-bb-8` (section gap) below the transactions card.

**Button spec:**
- Use existing `<SyncTransactionsButton accountCount={n} />` — unchanged component.
- Width: full-width (`w-full`) on the home page. Rationale: single primary action per view (DESIGN_SYSTEM §5.6 "max 1 primary button per view"); full-width reinforces that it is the page's primary action and provides a generous touch target (A11Y-01).
- The button internally uses shadcn `Button` with variant `default` (which maps to `--bb-info` blue fill in the new-york preset wired to BB tokens). No change to the component itself.

### Edge State: DB Error (Inline note)

**Hub still renders** with all 4 sections visible (CONTEXT D-CUT-02).

- Hero shows `€—` in `text-bb-text-secondary` as specified above.
- Budget and transaction sections render with empty states (0-item path).
- At the bottom of the page (below the Sync button), add one small paragraph: `<p className="text-bb-xs text-bb-text-secondary text-center mt-bb-4">Couldn't load data. Try refreshing.</p>`
- No red banner. No `--bb-negative` color. (DESIGN_SYSTEM §5.5 "max 1 banner" and P1 calm-over-comprehensive. CONTEXT D-CUT-02.)

## Section Dividers

**Between the 4 sections: `gap-bb-8` (32px) only.** No horizontal lines, no tinted backgrounds per section. The section cards themselves (`bg-bb-surface` + `border-bb-border` + `rounded-bb-lg`) provide sufficient visual separation. Adding dividers would increase visual noise — contradicts P1.

**Implementation:** Wrap the 4 sections in `<div className="flex flex-col gap-bb-8">`.

## Loading / Skeleton Policy

**No skeletons for Phase 8.**

The home page is a Next.js App Router Server Component (CONTEXT D-GR-04). All data fetches (`getUser`, `getAccounts`, `calculateAllBudgetProgress`, `getRecentTransactions`) complete server-side before the HTML is sent. First paint = final paint for data.

The only interactive island is `<SyncTransactionsButton>` (already a client component with its own loading state). That component's internal loading spinner is sufficient — no page-level skeleton needed.

**No layout shift on initial render.** ADHD_UX_RESEARCH.md §7.5 "layouts that shift during page load force constant re-orientation." Server rendering eliminates this.

If Next.js streaming ever splits this page into suspense boundaries in Phase 9+, a skeleton contract will be added at that time.

## Motion Policy

**No animations in Phase 8. This is a positive lock, not an oversight.**

- Phase 11 owns the motion system (MOTION-01, MOTION-02).
- `SyncTransactionsButton` handles its own click feedback internally — no Phase 8 change.
- Focus rings: use browser default + shadcn's built-in ring styles (`focus-visible:ring-2 focus-visible:ring-bb-info`). This is not motion — it is an a11y requirement (A11Y-01).
- No `transition`, `animation`, or `@keyframes` declarations in Phase 8 code.
